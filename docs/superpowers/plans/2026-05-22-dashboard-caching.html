Dashboard load-time caching

Goal Make warm dashboard loads near-instant and roughly halve the cold load by caching heavy read-endpoint responses and computing the deduped CTE once per request. Verbatim task steps live in the companion .md plan; this page is the readable overview.
For agentic workers: execute the .md plan task-by-task via superpowers:subagent-driven-development or superpowers:executing-plans. Each task is TDD: failing test → implementation → passing test → commit.

Architecture

Two independent changes. (1) A process-local TTL response cache plus a cache_response decorator on the six heavy read endpoints, flushed when ingest completes. (2) /api/dashboard builds the deduped set once into a ON COMMIT DROP temp table with a bumped work_mem, instead of re-running the CTE in five separate statements.

Tasks

Task 1 — TTL response cache + decorator

backend/cache.py · tests/test_cache.py
  1. _TTLCache: flat-TTL dict cache (get/put/clear), expiry on read.
  2. response_cache = _TTLCache(ttl_seconds=3600) as a safety-net TTL.
  3. cache_response: wraps an async endpoint, keys on its keyword args, follows functools.wraps so FastAPI still parses the real query params. A truthy fresh kwarg bypasses.

Task 2 — Apply the decorator

backend/api.py

@cache_response below @router.get(...) on: /dashboard, /cache, /tool-usage, /tool-error-rate, /reply-latency, /context-growth/agg. Behaviour test: a second call returns the cached payload even after the DB is mutated; fresh=1 bypasses.
