Export PNG button — render the dashboard for the active filters

Context The DB-backed plotting script scripts/plots/ccusage_plot_db.py already renders the full 8-panel matplotlib usage dashboard from the deduped, cost-priced records table — the same data the web dashboard shows. This spec wires it to a browser Export PNG button that honours the dashboard's active project + time-range filters, and walls the feature off from guests.

Decision
A new GET /api/export subprocesses the working script and streams back a PNG.
Subprocess over in-process: it reuses the script verbatim, keeps the heavy matplotlib dependency out of the app .venv, and isolates the CPU-bound render so it never blocks the uvicorn event loop. Matches the "since it works" intent.
Export is logged-in-only — guests get a 403 from the existing auth middleware and never see the button. It deliberately does not cache the PNG, queue async jobs, or honour the model filter (project + range only).
Post-review (commit 4768ab5): the DSN is not passed in argv — the subprocess inherits DATABASE_URL_VIZ from the env (keeps the password out of the process list); and the frontend uses fetch+blob with an .ok check rather than a plain <a href>, so error responses render inline instead of ejecting the SPA. The service runs in-place from /root/session-viz (the examples/ unit's /opt/ccudash is only a sample), so a restart needs no deploy-sync.
01

Background

What the reader needs to evaluate the rest.

The dashboard filters on range (1d/7d/30d/90d/365d/all) + project (a project_id that joins files) + model, deduping cross-file via DISTINCT ON (r.uuid) in /api/dashboard. ccusage_plot_db.py mirrors that exact dedup and reads the precomputed records.cost_usd, so its plot is data-consistent with the live dashboard by construction.

Two gaps remain: (a) the script has no project filter (its load_events is a bare FROM records r), and (b) nothing connects it to the web app. The deployed service runs from /opt/ccudash as root with EnvironmentFile=/opt/ccudash/.env, and the only interpreter with matplotlib is system /usr/bin/python3 — the app .venv does not have it.

02

Subprocess vs. in-process

Why the render runs in a separate process.

In-process module
  • Adds matplotlib to the app .venv — a heavy dep the web tier otherwise avoids.
  • pyplot global state + CPU-bound render risk blocking the event loop; needs threadpool care.
  • Requires refactoring the CLI script into an importable module.
Subprocess (chosen)
  • Reuses the working script verbatim (plus one --project arg).
  • matplotlib stays out of the app venv; render isolated in its own process.
  • asyncio.create_subprocess_exec never blocks the loop.
03

Data flow

From button click to downloaded PNG.

Browser ──GET /api/export?range=&project=──▶ FastAPI (backend/api.py)
                                                │ middleware: guest → 403 (session.py)
                                                │ arg map + asyncio.Semaphore(1)
                                                ▼
                    asyncio.create_subprocess_exec(   # child inherits DATABASE_URL_VIZ
                      /usr/bin/python3  scripts/plots/ccusage_plot_db.py
                      -o <tempfile>  [-p Nd | --all]  [--project P])
                                                │ matplotlib render, ≤120s
                                                ▼
                    read PNG bytes → unlink temp → Response(image/png, attachment)

The dashboard view already holds activeRange and activeProject in state and feeds them to /api/dashboard. The Export button forwards the same two values to /api/export; the script re-derives the identical deduped record set, so the PNG matches the on-screen panels.

04

Change inventory

Every file the implementation touches, and the precise change.

backend/api.py
New GET /api/export. Params range="30d", project=None (same as /api/dashboard). Maps all→--all, else -p <range>. Runs system /usr/bin/python3 on the script with -o <tempfile> and --project when set; the DSN is inherited from the environment (not passed in argv). Module-level asyncio.Semaphore(1) single-flights; wait_for(..., 120) → 503 on timeout, 500 on non-zero exit. Returns the PNG bytes as image/png with an attachment Content-Disposition; temp file unlinked. No @cache_response.
backend/session.py:180-183
Guest gate. Add path.startswith("/api/export") to the existing guest-blocked path check (beside /api/projects / /api/sessions) → guest gets 403 Forbidden (guest). One line; no new logic.
scripts/plots/ccusage_plot_db.py
Add --project. load_events(..., project=None) joins files and adds AND f.project_id = %(project)s when set; find_limit_hits() scoped the same way so burn-rate rate-limit markers match. No arg → byte-identical to today.
src/app.jsx
Export PNG button. In the top bar next to Logout, gated on backendOn && !isGuest. fetches /api/export?range=…&project=… and branches on res.ok — blob-download on success, inline error otherwise (so an error never ejects the SPA). A busy flag disables it during a render.
tests/test_api.py · (middleware test)
Tests. Monkeypatch the subprocess runner: assert argv for -p 7d / --all, --project presence/absence, --db-url absent (env-inherited DSN), image/png + attachment on success, and timeout→503 / exit→500. Guest 403 asserted at the middleware layer (the router-only harness bypasses auth).

Verification

How the implementer confirms the change is correct.

Guest 403 — middleware test rejects /api/export for a guest session.

08

Out of scope

Explicitly deferred — named so the reader knows it was considered, not missed.

Model filter in the export
Design covers project + range only, per request. Trivial to add later via the same arg-mapping path.
Async job queue / progress UI
Renders are seconds; single-flight + a brief disabled button suffice.
PNG response caching
Explicit user action, heavy binary, data changes each ingest. Revisit only if export traffic warrants it.
