Dashboard load-time caching

Problem The main index page takes ~4.4 s to render actual data. The /api/dashboard endpoint is the bottleneck. Goal: warm loads near-instant, cold loads roughly halved.

Root cause

/api/dashboard issues ~6 separate SQL statements, and every one re-runs the same deduped CTE — a DISTINCT ON (uuid) over records ⋈ files with the time-range filter. Measured on the live claude_viz DB (114k records, 42k in the 30d window):

ObservationDetail
CTE cost per run~220 ms
CTE runs per request~6 → ~1.3 s
Sort spills to diskexternal merge Disk: 3360kB — work_mem is only 4 MB
records_ts_idxexists but planner correctly ignores it (30d = 37% of rows → seq scan wins)
Caching alone leaves the first load after each hourly ingest at 4.4 s. Dashboard data only changes on ingest, so two complementary fixes are in scope.

Change 1 — Response cache warm loads

Cache the final JSON of the heavy read endpoints, keyed by (path, sorted query params). The data is global (not per-user), so a params-keyed cache is correct — auth middleware still gates access before the endpoint runs.

Endpoints cached

  • /api/dashboard, /api/cache, /api/tool-usage, /api/tool-error-rate, /api/reply-latency, /api/context-growth/agg
  • Light endpoints (/api/models, /api/projects, /api/me) are left uncached.

Mechanism

  • New response_cache in backend/cache.py beside the existing transcript _IdleLRU — a dict keyed by the request signature, storing the serialized JSON.
  • A small @cache_response decorator reads the request's query string to build the key.
  • fresh=1 on /api/dashboard bypasses the cache (already used as a browser cache-bust).
  • TTL ~1 h as a safety net; primary invalidation is event-driven.

Invalidation

ingest run completes └─ ingest.run_ingest() → events.broadcast_threadsafe("ingest_done") └─ NEW: response_cache.clear() └─ next dashboard load recomputes once, then warm again

The clear is wired at the same point that already broadcasts ingest_done (backend/ingest.py:234), so the cache is flushed exactly when — and only when — the underlying data changes.

Change 2 — Compute the CTE once cold loads

viz_conn() runs with autocommit=False, so each request is already a transaction. Within /api/dashboard:

SET LOCAL work_mem = '64MB'; -- kill the disk spill CREATE TEMP TABLE deduped ON COMMIT DROP AS -- run the CTE body ONCE ( ...existing deduped UNION ALL... ); -- each panel query now scans the 42k-row temp table (~10-20 ms)
  • The 6 panel queries change from base_cte + SELECT… to plain SELECT … FROM deduped.
  • ON COMMIT DROP is safe — the connection context manager commits on exit, dropping the temp table.
  • SET LOCAL scopes the work_mem bump to this one transaction.
Scope: Change 2 targets /api/dashboard only — it is the main index page and the user's actual complaint. The other deduped-pattern endpoints are covered by Change 1's response cache and are not restructured here.

Expected outcome

ScenarioBeforeAfter
Warm load (between hourly ingests) ~4.4 s ~instant (cache hit)
Cold load (first after an ingest) ~4.4 s ~2 s (CTE once + work_mem)

Testing

  • test_api.py: dashboard returns identical payload on a second call (cache hit); fresh=1 bypasses.
  • Cache cleared after a simulated ingest run — stale payload not served.
  • Existing dashboard correctness tests must pass unchanged (temp-table refactor is behaviour-preserving).

Out of scope

  • No persisted rollup table (banned by SV-FILES-RECORDS).
  • No HTTP Cache-Control/ETag headers for static assets — separate concern, not the data-load bottleneck.
  • No restructuring of /api/cache et al. internals.
